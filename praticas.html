Práticas Ágeis XP

Práticas fundamentais do Extreme Programming aplicadas ao contexto corporativo

O Extreme Programming (XP) define 12 práticas fundamentais que garantem qualidade técnica, comunicação efetiva e entrega contínua de valor. Estas práticas são aplicadas em diferentes fases do nosso processo.

Práticas de Planejamento

Planning Game

Processo colaborativo de planejamento onde cliente e desenvolvedores determinam o escopo de releases e iterações.

Benefícios:
  • Alinhamento entre expectativas técnicas e de negócio
  • Priorização baseada em valor real
  • Planejamento realista e flexível
  • Comunicação constante sobre progresso
Como aplicar:

Realize sessões de 2-4 horas por iteração com Customer e Developers, usando técnicas como Planning Poker para estimativas.

User Stories

Descrição de funcionalidades do ponto de vista do usuário, focando no valor entregue e não na implementação técnica.

Benefícios:
  • Foco no valor para o usuário final
  • Comunicação clara de requisitos
  • Facilita priorização e estimativas
  • Base para testes de aceitação
Formato padrão:

"Como [tipo de usuário], eu quero [funcionalidade] para que [benefício]"

Small Releases

Entregas frequentes de software funcionando em pequenos incrementos, tipicamente a cada 1-2 semanas.

Benefícios:
  • Feedback rápido do cliente
  • Redução de riscos
  • Demonstração contínua de progresso
  • Ajustes rápidos de direção
Meta:

Cada release deve conter 1-3 histórias de usuário completas e demonstráveis.

Práticas de Desenvolvimento

Pair Programming

Duas pessoas trabalhando no mesmo código simultaneamente, uma como "driver" (digitando) e outra como "navigator" (revisando).

Benefícios:
  • Qualidade de código superior
  • Compartilhamento de conhecimento
  • Redução de defeitos
  • Mentoria contínua
Rotação:

Alterne pares diariamente e troque papéis driver/navigator a cada 15-30 minutos.

Test-Driven Development (TDD)

Prática de escrever testes antes do código de produção, seguindo o ciclo Red-Green-Refactor.

Benefícios:
  • Design emergente e limpo
  • Cobertura de testes natural
  • Confiança para refatorar
  • Documentação viva do código
Ciclo TDD:

1. Escreva um teste que falhe (Red) → 2. Implemente o mínimo para passar (Green) → 3. Refatore (Refactor)

Refactoring

Melhoria contínua da estrutura interna do código sem alterar seu comportamento externo.

Benefícios:
  • Código mais limpo e maintível
  • Redução de dívida técnica
  • Facilita futuras mudanças
  • Melhora legibilidade
Quando refatorar:

Sempre que você vê código duplicado, métodos muito longos, ou quando adicionar nova funcionalidade é difícil.

Simple Design

Manter o design do sistema o mais simples possível, implementando apenas o que é necessário agora.

Benefícios:
  • Desenvolvimento mais rápido
  • Menos complexidade
  • Facilita mudanças
  • Reduz over-engineering
Princípios:

1. Passa todos os testes → 2. Não tem duplicação → 3. Expressa intenção → 4. Mínimo de classes/métodos

Collective Code Ownership

Todos os desenvolvedores têm responsabilidade sobre todo o código do sistema.

Benefícios:
  • Conhecimento distribuído
  • Reduz gargalos
  • Melhoria contínua do código
  • Responsabilidade coletiva
Como implementar:

Qualquer desenvolvedor pode modificar qualquer código, mas sempre com pair programming ou revisão.

Coding Standards

Convenções de codificação consistentes em toda a equipe para facilitar compreensão e manutenção.

Benefícios:
  • Código uniforme e legível
  • Facilita pair programming
  • Reduz tempo de compreensão
  • Melhora qualidade geral
Elementos:

Nomenclatura, formatação, estrutura de arquivos, comentários e documentação inline.

Práticas de Testes

Acceptance Tests

Testes automatizados que verificam se o sistema atende aos critérios de aceitação definidos pelo cliente.

Benefícios:
  • Validação automática de requisitos
  • Documentação executável
  • Confiança nas entregas
  • Comunicação clara de expectativas
Ferramenta sugerida:

Use frameworks como Cucumber, Selenium ou Jest para automatizar cenários de aceitação.

Continuous Integration

Integração frequente do código da equipe com execução automática de testes e verificações de qualidade.

Benefícios:
  • Detecção precoce de problemas
  • Código sempre integrado
  • Feedback rápido
  • Reduz riscos de integração
Regra de ouro:

Nunca deixe o build quebrado por mais de 10 minutos. Se quebrar, pare tudo e conserte.

Práticas de Gerência

On-site Customer

Cliente disponível em tempo integral para a equipe, esclarecendo dúvidas e fornecendo feedback imediato.

Benefícios:
  • Feedback imediato
  • Decisões rápidas
  • Reduz mal-entendidos
  • Priorização dinâmica
Adaptação:

No contexto acadêmico, pode ser o professor, cliente real ou proxy representando stakeholders.

Sustainable Pace

Manter ritmo de trabalho sustentável, evitando burnout e mantendo qualidade consistente.

Benefícios:
  • Qualidade consistente
  • Evita burnout
  • Produtividade sustentável
  • Melhor moral da equipe
Meta:

40 horas por semana é o máximo. Horas extras devem ser exceção, não regra.

Metaphor

Visão compartilhada do sistema através de uma metáfora que todos entendem, facilitando comunicação.

Benefícios:
  • Comunicação efetiva
  • Visão arquitetural comum
  • Facilita design decisions
  • Nomenclatura consistente
Exemplo:

Sistema de e-commerce como "loja virtual" - com carrinho, caixa, estoque, vendedor, etc.

Guia de Implementação por Iteração

1

Primeira Iteração

Foque em: Planning Game, User Stories, Simple Design, Coding Standards

Estabeleça a base do processo e acerte as práticas fundamentais antes de avançar.

2

Segunda Iteração

Adicione: Pair Programming, TDD básico, Small Releases

Introduza práticas de desenvolvimento colaborativo e comece com testes.

3

Terceira Iteração

Aprimore: TDD avançado, Refactoring, Acceptance Tests

Refine as práticas técnicas e introduza testes de aceitação automatizados.

4+

Iterações Seguintes

Domine: Continuous Integration, Collective Ownership, Sustainable Pace

Integre todas as práticas e foque na melhoria contínua do processo.

Checklist de Verificação

Planejamento

  • □ Histórias escritas no formato padrão
  • □ Estimativas feitas pela equipe toda
  • □ Priorização clara do cliente
  • □ Planning game realizado
  • □ Release pequena definida

Desenvolvimento

  • □ Pair programming sendo praticado
  • □ TDD sendo seguido
  • □ Refatoração contínua
  • □ Design simples mantido
  • □ Padrões de código seguidos
  • □ Propriedade coletiva do código

Testes

  • □ Testes unitários passando
  • □ Testes de aceitação definidos
  • □ Integração contínua funcionando
  • □ Cobertura de testes adequada

Gerência

  • □ Cliente disponível para equipe
  • □ Ritmo sustentável mantido
  • □ Metáfora do sistema clara
  • □ Comunicação efetiva
